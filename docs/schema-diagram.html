시계열 분석용 (선택적)
✅ 최종 구조: profiles + asset_groups + asset_group_members + snapshots = 실 테이블 4개
한 유저가 여러 장부 참여 가능 (개인 장부 + 계 장부). 스냅샷은 장부에 귀속.
역할 권한 (Role-based Access)

멤버 초대 / 역할 변경 ✓✗✗

RLS 동작: 권한 검사는 DB 레벨에서 asset_group_members를 조회해 처리.
비멤버는 장부/스냅샷 자체를 조회할 수 없음 (404가 아닌 빈 결과 반환).
시나리오 예시
개인 자산 관리
# 가입 시 자동 생성 asset_groups: { name: "내 자산", type: "personal" } asset_group_members: { role: "owner" } # 혼자 스냅샷 작성/수정 # viewer/editor 초대 불필요
계 (공동 장부)
# 계장이 장부 생성 asset_groups: { name: "2026 목돈 계", type: "group" } asset_group_members: { role: "owner" } # 계장 # 계원 초대 asset_group_members: { role: "editor" } # 입력 권한 asset_group_members: { role: "viewer" } # 열람만 # 계원들이 함께 스냅샷 수정
가입 시 자동 흐름
신규 가입 → trigger
-- Supabase auth.users INSERT 시 trigger 실행 BEGIN -- 1. 프로필 생성 INSERT INTO profiles(id) VALUES (NEW.id); -- 2. 기본 개인 장부 생성 INSERT INTO asset_groups(name, type, created_by) VALUES ('내 자산', 'personal', NEW.id) RETURNING id INTO v_group_id; -- 3. 해당 장부의 owner로 등록 INSERT INTO asset_group_members(group_id, user_id, role) VALUES (v_group_id, NEW.id, 'owner'); END;
